# Codegen Integration

**Codegen repo:** `github.com/cdoan1/hyperfleet-api-codegen`
**Jira:** [ROSAENG-61801](https://redhat.atlassian.net/browse/ROSAENG-61801)
**Parent:** [ROSAENG-61383](https://redhat.atlassian.net/browse/ROSAENG-61383)

## Overview

The codegen system generates typed Go structs, field metadata, and OpenAPI schemas from HyperShift upstream CRD types. It enforces field-level access control (write-mode, feature gates, visibility) at both compile time (typed structs) and runtime (validation middleware).

## Architecture

| Component | Location | Purpose |
|-----------|----------|---------|
| **API types** | `api/v2alpha1/` | Go structs annotated with marker comments |
| **Runtime libraries** | `internal/codegen/` | Registry, feature gates, validation, conversion |
| **Generator tools** | Codegen repo, installed as binaries in `./bin/` | `passthrough-gen`, `marker-scanner`, `openapi-gen` |
| **OpenAPI specs** | `openapi/` | Generated schemas and hand-written API spec |
| **Merge script** | `hack/merge-openapi.sh` | Patches generated schemas into `openapi.yaml` |

```
    │
    ├──▶ marker-scanner ──▶ internal/codegen/registry/field_metadata.go
    │                       internal/codegen/registry/field_metadata.json
    │                           │
    │                           ▼
    │                       validation.Validator ──▶ middleware.FieldValidator
    │                                                    │
    │                                                    ▼
    │                                               HTTP handlers

    └──▶ openapi-gen ──▶ openapi/generated-schemas.json

```

## Marker Annotations

Three families of Go comment markers control field behavior:

### `+hyperfleet:write-mode=<value>`

Controls field mutability at runtime.

| Value | Meaning |
|-------|---------|
| `mutable` | Customer can set on create and change on update |
| `immutable` | Customer can set on create but cannot change on update |
| `service-set` | Platform-managed — customers cannot set it |

### `+k8s:openapi-gen=<bool>`

Controls OpenAPI visibility.

| Value | Meaning |
|-------|---------|
| `true` | Explicitly include in generated OpenAPI schemas |
| `false` | Exclude from generated OpenAPI schemas (field is hidden) |

### `+openshift:enable:FeatureGate=<GateName>`

Gates a field behind a named feature gate. The field is rejected at runtime unless the request's `FeatureSet` includes the gate's maturity stage.

### Example

```go
// Tags are customer-defined labels for organizational purposes
// +hyperfleet:write-mode=mutable
// +openshift:enable:FeatureGate=HyperFleetAutoScaling
Tags map[string]string `json:"tags,omitempty"`
```

## Type System

### ClusterSpec structure

`ClusterSpec` has two sections:

1. **Envelope fields** — HyperFleet-specific platform fields:
   - `DisplayName`, `DeleteProtection`, `ExpirationTimestamp`, `Properties`, `Tags` (customer-facing)
   - `CloudUrl`, `Placement`, `CreatorARN`, `AccountID`, `InternalID` (platform-managed)

2. **Passthrough embed** — `HostedCluster *HostedClusterSpecPassthrough`:
   - Generated mirror of `hypershiftv1beta1.HostedClusterSpec`
   - All fields default to `+k8s:openapi-gen=false` and `+hyperfleet:write-mode=service-set`
   - Fields are selectively exposed by changing markers during curation

### Passthrough curation workflow

`passthrough-gen` produces a raw Go file mirroring upstream HyperShift types. The developer then curates marker annotations on each field to control visibility and write-mode. The raw output is kept as `zz_generated.passthrough.go.raw` for reference; the curated version is `hostedclusterspec.passthrough.go`.

### NodePoolSpec

Same pattern: envelope fields (`ClusterRef`, `DisplayName`, `AutoRepair`, etc.) plus `NodePool NodePoolSpecPassthrough` embed.

### Configuration types

`configuration.go` provides HyperFleet-owned mirrors of HyperShift configuration types with granular per-field markers. This enables fine-grained control over which nested configuration fields customers can set (e.g., `kubeletConfig.maxPods` is mutable, `kubeletConfig.evictionHard` is hidden/service-set).

## Feature Gate System

Defined in `internal/codegen/featuregate/`.

### Maturity stages

| Stage | Constant | Included in |
|-------|----------|-------------|
| GA | `featuregate.GA` | All feature sets |
| TechPreview | `featuregate.TechPreview` | `TechPreviewNoUpgrade`, `DevPreviewNoUpgrade` |
| DevPreview | `featuregate.DevPreview` | `DevPreviewNoUpgrade` only |

### Feature sets

| Set | Fields available |
|-----|-----------------|
| `Default` | GA only |
| `TechPreviewNoUpgrade` | GA + TechPreview |
| `DevPreviewNoUpgrade` | GA + TechPreview + DevPreview |

### Registered gates

| Gate | Stage | Description |
|------|-------|-------------|
| `HyperFleetEtcdConfig` | GA | Etcd settings |
| `HyperFleetAutoScaling` | TechPreview | Cluster autoscaling |
| `HyperFleetSecretEncryption` | TechPreview | Secret encryption |
| `HyperFleetKubeletAdvanced` | TechPreview | Advanced kubelet config |
| `HyperFleetMachineConfig` | TechPreview | Kernel parameter allowlisting |
| `HyperFleetCustomDNS` | DevPreview | Custom DNS configuration |

### FeatureGateAwareWriteModes

A field's write-mode can vary based on enabled gates. For example, a field might be `immutable` by default but `mutable` when a premium gate is enabled. The first matching gate-specific override wins, then the default (empty gate) override, then the base `WriteMode`.

## Validation System

### Flow

1. HTTP handler JSON-decodes request body into typed `*v2alpha1.ClusterSpec`
2. Handler calls `conversion.SpecToMap(req.Spec)` to produce `map[string]any`
3. Handler passes map to `middleware.FieldValidator.ValidateCreate()` or `ValidateUpdate()`
4. `FieldValidator` calls `flattenWithPrefix("spec", specMap)` to convert nested maps to dotted paths (e.g., `spec.displayName`, `spec.hostedCluster.channel`)
5. `validation.Validator.Validate()` checks each field path against `FieldRegistry`:
   - **Write-mode**: `service-set` fields rejected; `immutable` fields rejected on update if changed
   - **Feature gate**: Field rejected if gate's stage exceeds the request's `FeatureSet`
6. Returns `ValidationErrors` (HTTP 422) or nil (proceed)

### Error response format

```json
{
  "kind": "Error",
  "code": "CLUSTERS-MGMT-VALIDATE-001",
  "reason": "Validation failed",
  "details": [
    {"field": "spec.accountId", "reason": "field is platform-managed (service-set) and cannot be set by customers"}
  ]
}
```

## Conversion Layer

`internal/codegen/conversion/` bridges typed Go structs and the map-based interfaces used by the validator and Hyperfleet wire protocol.

### Generic bridge functions (`specmap.go`)

| Function | Direction | Used at |
|----------|-----------|---------|
| `SpecToMap(spec any) (map[string]any, error)` | Typed struct → map | Validator boundary (handlers), Hyperfleet wire requests |
| `MapToSpec[T any](m map[string]any) (*T, error)` | Map → typed struct | Hyperfleet wire responses |

Both use JSON round-trip (`json.Marshal` → `json.Unmarshal`).

### Cluster-specific functions (`cluster.go`)

| Function | Purpose |
|----------|---------|
| `InjectClusterServiceSet(spec *ClusterSpec, ssf)` | Merges platform-injected values (`CloudURL`, `Placement`, `CreatorARN`) into typed spec during create |
| `RewriteCloudURLWithID(spec *ClusterSpec, baseURL, clusterID)` | Sets `CloudUrl` to `baseURL/clusterID` in response enrichment |

### Boundary diagram

```
Typed Go structs (*v2alpha1.ClusterSpec)
    │
    ├──▶ SpecToMap ──▶ FieldValidator.ValidateCreate/Update (maps with dotted paths)
    │
    ├──▶ SpecToMap ──▶ HFClusterCreateRequest.Spec (wire map to Hyperfleet)
    │
    └──◀ MapToSpec ──◀ HFCluster.Spec (wire map from Hyperfleet)
```

The Hyperfleet wire types (`HFCluster`, `HFClusterCreateRequest`, etc. in `pkg/clients/hyperfleet/types.go`) remain `map[string]interface{}` because the Hyperfleet API is schema-agnostic.

## OpenAPI

### `--keep-markers` development mode

Pass `KEEP_MARKERS=1` to `make codegen-openapi` to:

- Preserve marker annotation strings in schema descriptions (visible in Swagger UI)
- Include hidden passthrough objects (`hostedCluster`, `nodePool`) in schemas
- Import all generated definitions as standalone `components.schemas` entries
- Rewrite `$ref` paths from Swagger 2.0 (`#/definitions/X`) to OpenAPI 3.0 (`#/components/schemas/X`)

Without `--keep-markers`, marker lines are stripped and passthrough objects are filtered out (production mode).

### Schema patching

`merge-openapi.sh` extracts `ClusterSpec` and `NodePoolSpec` properties from `generated-schemas.json` and patches them into `openapi.yaml`:

- `ClusterSpec` created as a standalone schema under `components.schemas`
- `Cluster.spec` uses `$ref: '#/components/schemas/ClusterSpec'`
- `ClusterCreateRequest.spec` and `ClusterUpdateRequest.spec` use `allOf` with `$ref`
- Server-enriched fields (`cloudUrl`, `placement`) are added to the schema

## Generated vs Hand-Written Files

Generated files carry the standard header: `// Code generated by <tool>. DO NOT EDIT.`

| File | Status |
|------|--------|
| `api/v2alpha1/hostedclusterspec.passthrough.go` | Generated by `passthrough-gen`, then manually curated |
| `api/v2alpha1/zz_generated.passthrough.go.raw` | Generated reference copy (not compiled) |
| `internal/codegen/registry/field_metadata.go` | Generated by `marker-scanner` |
| `internal/codegen/registry/field_metadata.json` | Generated by `marker-scanner` |
| `openapi/generated-schemas.json` | Generated by `openapi-gen` |
| Everything else | Hand-written |

## Workflows

### HyperShift version bump

```bash
go get github.com/openshift/hypershift/api@<new-commit>
go mod tidy

make codegen-passthrough        # Regenerate passthrough types
# Review hostedclusterspec.passthrough.go — add markers to new fields

make codegen-registry           # Regenerate field metadata
make codegen-openapi            # Regenerate OpenAPI schemas
make codegen-verify
make test
```

### Adding or changing a field marker

```bash
# Edit api/v2alpha1/hostedclusterspec.passthrough.go (or other type files)
# Add/change markers: +hyperfleet:write-mode=mutable, +openshift:enable:FeatureGate=..., etc.

make codegen-registry           # Regenerate field metadata
make codegen-openapi            # Regenerate OpenAPI schemas
make codegen-verify
make test
```

### Adding a new envelope field to ClusterSpec

```bash
# Edit api/v2alpha1/cluster_types.go — add the field with markers

make codegen-registry           # Picks up the new field
make codegen-openapi KEEP_MARKERS=1  # Verify in Swagger UI
make codegen-verify
make test
```

### Adding a new feature gate

1. Add gate to `internal/codegen/featuregate/registry.go` in `HyperFleetFeatureGates`
2. Annotate fields with `+openshift:enable:FeatureGate=<NewGateName>` in `api/v2alpha1/`
3. Run `make codegen-registry` and `make test`

### Full regeneration from scratch

```bash
make codegen-install-tools      # Install all tool binaries
make codegen-passthrough        # Generate passthrough types from HyperShift
# Curate markers in hostedclusterspec.passthrough.go
make codegen-registry           # Generate field metadata registry
make codegen-openapi            # Generate and merge OpenAPI schemas
make codegen-verify             # Verify compilation
make test                       # Run tests
```
